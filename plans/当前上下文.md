# Momo 当前压缩上下文

更新时间：2026-05-16

## 已完成

- 仓库在 `main` 直接开发，用户要求每阶段验证可运行后提交，后续提交信息使用中文。
- 桌面端位于 `apps/desktop`，技术栈为 Tauri 2 + React 19 + TypeScript + Vite。
- 已接入 `@tauri-apps/plugin-sql` / `tauri-plugin-sql`，SQLite 连接固定为 `sqlite:momo.db`。
- 已实现 `TaskRepository`，页面不直接写 SQL。
- 已实现本地任务 MVP：
  - `Today`：快速添加今日任务，显示逾期、今日、今日完成。
  - `Inbox`：显示无截止日期任务，支持编辑、完成、删除。
  - `Calendar`：显示未来 7 天只读 agenda。
  - `Settings`：显示本地数据库状态。
- 已清理 Vite 模板遗留，README 已更新。
- 已实现桌面 Widget 壳：
  - `/widget` 紧凑路由。
  - 主窗口侧栏 `Open widget` 入口。
  - `widget` 窗口获得 SQL 与窗口权限。
  - 当前只做 Tauri WebviewWindow 创建/聚焦，不做 Win32 扩展样式桥接。
- 已补全基础任务字段体验：
  - Today 快速添加可选择 Today 或 Inbox。
  - Inbox 编辑支持 title、notes、priority。
- 已实现同步前本地操作日志：
  - SQLite 增加 `local_changes` 表。
  - `TaskRepository` 对 create/update/status/delete 追加本地变更记录。
  - 新增 `listPendingChanges()` 与 `markChangeSynced()`，为后续 Delta Sync 铺路。
- 已补全 dueAt / estimateMin 基础编辑：
  - Today 快速添加支持显式截止时间和估时分钟。
  - Inbox 编辑支持 dueAt 和 estimateMin。
- 已建立同步/API contract 骨架：
  - 新增 `packages/contracts` 共享 TypeScript 包。
  - 定义 Task DTO、LocalChange DTO、Delta Push/Pull 请求响应类型。
  - 新增 delta push/pull 请求构造函数，固定 `SYNC_CONTRACT_VERSION = 1`。
  - 桌面端测试覆盖 contract payload 形状，contract 包可独立 `tsc` 编译。
- 已完成本地 MVP 页面验收硬化：
  - Inbox / Calendar / Settings 的加载错误都提供可键盘操作的 Retry。
  - 错误状态不再和空状态同时显示。
  - Inbox 编辑支持把已有 dueAt / estimateMin 清空为 `null`。
  - Inbox 完成、删除、保存失败会留在页面并显示错误。
- 已完成最终桌面端 MVP 收束核查：
  - README 已同步当前 schema、`/widget` 路由、contract 包与验证命令。
  - 原始本地 MVP 范围已落地：SQLite、本地 Repository、Today / Inbox / Calendar / Settings、widget 壳、测试与文档。
  - 最终验证通过：`npm run test`、`npm run build`、`cargo check`、contract 包 `tsc`。
- 已固化持续开发循环原则：
  - 每阶段完成后必须更新本文件，压缩本阶段结果、验证命令、下一阶段目标和阶段循环原则。
  - 每阶段验证通过后使用中文提交信息提交，然后继续下一阶段。
- 已建立最小 API / Sync 服务骨架：
  - 新增 `apps/api` 纯 TypeScript 包。
  - 新增 `createSyncApi()` 与 `createInMemorySyncStore()`，支持 delta push / pull 的内存语义验证。
  - 当前只接共享 contract，不启动 HTTP server，不接 OIDC、PostgreSQL、Redis、WebSocket 或生产环境。
  - 桌面 Vitest 覆盖 push/pull 与不支持 contract version 的拒绝路径。
- 已补齐 Sync 服务本地语义覆盖：
  - 支持 `task.update`、`task.status`、`task.delete` 本地应用。
  - 支持基于 `cursor-N` 的增量 pull。
  - 同一 local change id 重放保持幂等，不重复推进 cursor。
  - 被拒绝的变更返回拒绝原因且不推进 server cursor。
- 已建立 Core API 任务 CRUD 服务骨架：
  - `apps/api/src/tasks.ts` 新增纯 TypeScript `TaskService` 与 `TaskRepository` 接口。
  - 新增 `createInMemoryTaskRepository()` 内存实现。
  - 支持 create/update/status/delete/list，复用 `TaskDto`。
  - 新增 `TaskActor` 权限占位参数，viewer 写入会被拒绝，workspace list 隔离有测试覆盖。
- 已建立 API 路由适配层骨架：
  - `apps/api/src/router.ts` 新增 HTTP-like `createApiRouter()`。
  - 支持 `/tasks`、`/sync/delta/push`、`/sync/delta/pull` 的纯函数请求分派。
  - 覆盖 JSON parse、状态码、错误响应、actor headers 注入占位。
  - 仍不启动真实 server，不接 OIDC / PostgreSQL / WebSocket。
- 已建立仓库级本地验证入口：
  - 根目录新增 `package.json`。
  - 新增 `npm run verify`，串行执行 desktop test/build、Tauri cargo check、contracts tsc、api tsc。
  - 新增测试确保验证脚本不会漂移。
- 已建立 Sync 冲突对象 contract 骨架：
  - `packages/contracts` 新增 `TaskConflictDto` 与 `createTaskConflict()`。
  - `DeltaPushResponse` 新增 `conflicts` 字段。
  - 当前 `apps/api` 内存 sync 在无冲突场景返回 `conflicts: []`。
- 已实现 Sync 最小冲突检测：
  - `apps/api` 内存 sync 支持 update/status payload 的 `baseVersion` 检测。
  - 客户端基于旧版本修改任务时返回 `TaskConflictDto`。
  - 冲突变更不进入 accepted/rejected，不应用变更，不推进 cursor。
- 已建立 Sync 冲突解决 contract/API 骨架：
  - `packages/contracts` 新增 `ResolveTaskConflictRequest/Response` 与 `createResolveTaskConflictRequest()`。
  - `apps/api` 的 `SyncApi` 新增 `resolveConflict()`。
  - 内存 sync 会保存检测到的 conflict，`server_wins` 可标记 resolved 并返回当前 server task。
  - `client_wins` / `manual` 目前作为 pending manual 占位，不做复杂 merge。
- 已把冲突解决接入 API 路由适配层：
  - `apps/api/src/router.ts` 支持 `POST /sync/conflicts/resolve`。
  - route 测试覆盖成功解决冲突和 missing conflict 的 404 错误响应。
- 已固化 API route manifest 与本地 runbook：
  - `apps/api/src/router.ts` 导出 `API_ROUTES`，测试覆盖全部 HTTP-like endpoint。
  - `apps/api/README.md` 记录当前 routes、actor header 占位、验证命令和明确限制。
- 已建立桌面端同步请求构造适配层：
  - `apps/desktop/src/sync/syncClient.ts` 新增 `buildDeltaPushFromPendingChanges()`。
  - 可从 `TaskRepository.listPendingChanges()` 构造 `DeltaPushRequest`。
  - 测试覆盖 LocalChange -> LocalChangeDto 映射、空队列、workspace/device 参数。
- 已建立桌面端同步响应处理适配层：
  - `apps/desktop/src/sync/syncClient.ts` 新增 `applyDeltaPushResponse()`。
  - accepted change ids 会调用 `TaskRepository.markChangeSynced()`。
  - rejected changes / conflicts 暂作为摘要返回给上层，不自动解决。
- 已在 Settings 暴露本地同步队列状态：
  - `DatabaseStats` 新增 `pendingLocalChanges`。
  - `TaskRepository.getStats()` 统计 `local_changes.synced_at IS NULL`。
  - Settings 的 Local database 卡片显示 `Pending sync`。
- 已补齐本地同步文档与验收说明：
  - `apps/desktop/README.md` 记录 `Pending sync`、同步请求/响应适配层和 `npm run verify`。
  - 新增 README 回归测试，防止本地同步说明漂移。

## 当前验证命令

在 `apps/desktop`：

```bash
npm run test
npm run build
```

在 `apps/desktop/src-tauri`：

```bash
cargo check
```

如本阶段触碰 `packages/contracts`，还需在仓库根目录运行：

```bash
.\apps\desktop\node_modules\.bin\tsc.cmd -p packages\contracts\tsconfig.json
```

如本阶段触碰 `apps/api`，还需在仓库根目录运行：

```bash
.\apps\desktop\node_modules\.bin\tsc.cmd -p apps\api\tsconfig.json
```

阶段提交前必须跑完相关命令并确认 exit 0。

## 开发规范

- 每阶段修改前先确认 `git status --short --branch`。
- 每阶段先写失败测试，再实现；测试文件优先放在 `apps/desktop/tests`。
- 每阶段验证通过后立即提交；提交信息用中文。
- 持续按路线图推进：每完成一个阶段，必须更新 `plans/当前上下文.md`，把本阶段结果、验证命令、下一阶段目标和本条阶段循环原则压缩进去，再提交，然后继续下一阶段。
- 任何对话上下文压缩都必须保留上一条原则：阶段完成后更新当前上下文、验证、中文提交、继续下一阶段。
- 不接后端时，业务数据只走 `TaskRepository`，不要在页面直接写 SQL。
- 保留 Rust `greet` smoke test；除 Tauri 插件/窗口能力外，不把业务逻辑搬到 Rust。
- 不重写历史，不回滚用户或已提交改动。
